Q3 — Building Blocks of Angular

Building Blocks of Angular

Modules

Group related components, directives, pipes, services. Root module = AppModule. Feature modules enable lazy loading.

@NgModule({
  declarations: [AppComponent],
  imports: [BrowserModule],
  bootstrap: [AppComponent]
})
Components

The UI building blocks. Each has a template, styles, and logic. Decorated with @Component.

@Component({
  selector: 'app-root',
  template: `<h1>Hello</h1>`
})
Templates

HTML extended with Angular syntax — interpolation {{}}, directives, event binding, two-way binding.

Services

Singleton classes for shared logic, API calls, state. Decorated with @Injectable. Injected via DI.

Directives

Extend HTML behaviour. Structural (*ngIf, *ngFor) and Attribute (ngClass, custom). No template.

Pipes

Transform data in templates. Built-in: date, currency, async. Custom pipes with @Pipe.

Routing

Maps URLs to components. Supports lazy loading, guards, resolvers, child routes.

DI System

Angular's built-in IoC container. Services are provided and injected automatically.

RxJS

Reactive programming library. Powers HTTP client, forms, event streams. Core to Angular's async model.

Q6 — Decorators in Angular

Decorators in Angular

Decorators are TypeScript metadata annotations prefixed with @. They tell Angular how to process a class.

Class Decorators 
@Component({ selector, template, styles })
@NgModule({ declarations, imports, providers })
@Injectable({ providedIn: 'root' })
@Pipe({ name: 'myPipe' })
@Directive({ selector: '[myDir]' })
Property Decorators 
@Input()  title: string;         // receive from parent
@Output() clicked = new EventEmitter();
@ViewChild('myRef') el: ElementRef;
@ContentChild(MyDir) dir: MyDir;
@HostBinding('class.active') isActive: boolean;
Method Decorators 
@HostListener('click', ['$event'])
onClick(event: MouseEvent) {
  // handle click on host element
}
Parameter Decorators 
constructor(
  @Inject(PLATFORM_ID) private platformId: object,
  @Optional() private config: AppConfig
) {}
💡 Senior Insight: Decorators are compiled away at build time — they're pure TypeScript metadata. @Injectable({ providedIn: 'root' }) is the modern way to register services — no need to add to module providers array.
Q7 + Q8 — Directives & Component vs Directive

Directives

3 Types of Directives

Component Directive

A directive with a template. The most common type.

Structural Directive

Changes DOM structure. Prefixed with *.

*ngIf="condition"
*ngFor="let item of items"
*ngSwitch, @if, @for (v17+)
Attribute Directive

Changes appearance or behaviour. No DOM change.

ngClass, ngStyle
Custom: [highlight], [tooltip]

Custom Attribute Directive

@Directive({ selector: '[appHighlight]' })
export class HighlightDirective {
  constructor(private el: ElementRef) {}

  @HostListener('mouseenter')
  onEnter() { this.el.nativeElement.style.background = 'yellow'; }

  @HostListener('mouseleave')
  onLeave() { this.el.nativeElement.style.background = ''; }
}

Component vs Directive

FeatureComponentDirective
Template✅ Yes❌ No
Styles✅ Yes❌ No
Decorator@Component@Directive
SelectorElement tagAttribute / CSS
Use caseUI blockBehaviour / DOM manipulation
LifecycleFull setSubset (no view hooks)
💡 Senior Insight: A Component is a special type of Directive. Internally Angular treats them similarly — @Component extends @Directive.
⚡ Angular 17+ Control Flow: New @if, @for, @switch syntax replaces *ngIf / *ngFor directives for better performance and readability.
Q9 — Component Communication

Component Communication Methods

1. @Input / @Output (Parent ↔ Child) 
// Parent → Child
@Input() title: string;

// Child → Parent
@Output() selected = new EventEmitter<number>();
this.selected.emit(userId);
2. ViewChild / ContentChild 
// Access child component directly
@ViewChild(ChildComponent) child!: ChildComponent;

ngAfterViewInit() {
  this.child.doSomething();
}
3. Shared Service (Any components) 
@Injectable({ providedIn: 'root' })
export class SharedService {
  private state$ = new BehaviorSubject(null);
  state = this.state$.asObservable();
  setState(val: any) { this.state$.next(val); }
}
4. NGRX Store (Large apps) 
// Dispatch action
this.store.dispatch(loadUsers());

// Select state
users$ = this.store.select(selectAllUsers);
5. Router State / Query Params 
// Send via navigation
this.router.navigate(['/user'], {
  queryParams: { id: 42 }
});

// Receive
this.route.queryParams.subscribe(p => p.id);
6. Local / Session Storage (Global) 
// For persisted cross-tab state
localStorage.setItem('token', jwt);
localStorage.getItem('token');
Q10 — Observable vs Promise vs Async/Await

Observable vs Promise vs Async/Await

FeaturePromiseObservable (RxJS)Async/Await
ValuesSingle valueMultiple values (stream)Single value
Lazy?❌ Eager (runs immediately)✅ Lazy (runs on subscribe)❌ Eager
Cancellable?❌ No✅ Yes (unsubscribe)❌ No (AbortController needed)
Operators?Limited (.then, .catch)✅ 100+ operatorsLimited (try/catch)
Retry?Manual✅ retry(3)Manual loop
Angular HTTPNeeds toPromise()✅ NativeVia firstValueFrom()
Use caseOne-time async opStreams, events, HTTPSequential async
// Observable — HTTP call (Angular native)
this.http.get<User[]>('/api/users')
  .pipe(
    retry(3),
    catchError(err => throwError(() => err))
  )
  .subscribe(users => this.users = users);
// Async/await with Observable
import { firstValueFrom } from 'rxjs';

async loadUsers() {
  const users = await firstValueFrom(
    this.http.get<User[]>('/api/users')
  );
  this.users = users;
}
Q11 — Lazy Loading

Lazy Loading

Load feature modules on demand when the route is first visited — drastically reduces initial bundle size.

Setup in app-routing.module.ts

const routes: Routes = [
  {
    path: 'dashboard',
    // Loaded only when user visits /dashboard
    loadChildren: () =>
      import('./features/dashboard/dashboard.module')
        .then(m => m.DashboardModule)
  },
  {
    path: 'admin',
    // Standalone component lazy loading (Angular 14+)
    loadComponent: () =>
      import('./admin/admin.component')
        .then(m => m.AdminComponent)
  }
];
✅ Result: Initial bundle only contains the code needed for the landing page. Other modules download in background or on demand.

PreloadingStrategy — Best of Both Worlds

// Preload all lazy modules after initial load
@NgModule({
  imports: [RouterModule.forRoot(routes, {
    preloadingStrategy: PreloadAllModules
  })]
})

Benefits

⚡ Faster initial load (smaller bundle)
📦 Code-split by route automatically
🔒 Admin features only load for admin users
📊 Lighthouse score improvement
⚠️ Interview Tip: Mention that you used lazy loading in production to reduce initial bundle from e.g. 2MB to 400KB, improving LCP (Largest Contentful Paint).
Q12–Q16 — Services, DI & Singleton

Services, Dependency Injection & Singleton

What is a Service?

A TypeScript class decorated with @Injectable that encapsulates reusable logic — API calls, state, utilities.

@Injectable({ providedIn: 'root' }) // Singleton ✓
export class UserService {
  constructor(private http: HttpClient) {}

  getUsers(): Observable<User[]> {
    return this.http.get<User[]>('/api/users');
  }
}

How DI Works

  1. Angular's Injector maintains a registry of providers
  2. When a component requests a service in its constructor, Angular looks it up
  3. If found, injects the existing instance (singleton)
  4. If not found, creates a new instance and registers it

Singleton vs Non-Singleton

// Singleton — one instance app-wide
@Injectable({ providedIn: 'root' })

// Non-singleton — new instance per component
@Component({
  providers: [UserService] // provided at component level
})

InjectionToken (non-class DI)

export const API_URL = new InjectionToken<string>('apiUrl');

// Provide
{ provide: API_URL, useValue: 'https://api.myapp.com' }

// Inject
constructor(@Inject(API_URL) private apiUrl: string) {}
💡 Senior Insight: "How does a service know which component to use?" — It doesn't. DI is inversion of control — the component declares what it needs; Angular's injector provides it. Services don't know their consumers.
Q19 + Q20 — BehaviourSubject & Observables

BehaviourSubject & Observables Deep Dive

Observable vs Subject vs BehaviourSubject

ObservableSubjectBehaviourSubject
Initial value❌❌✅ Required
Last value on subscribe❌❌✅ Yes
Can push values❌✅✅
Multicast❌✅✅
Use caseHTTP/eventsEventsState management
💡 When to use BehaviourSubject: Shared state between components — user auth state, cart count, theme toggle, language selection. Any state that needs a current value immediately on subscription.

BehaviourSubject State Service Pattern

@Injectable({ providedIn: 'root' })
export class AuthService {
  private user$ = new BehaviorSubject<User | null>(null);

  // Expose as Observable (read-only)
  currentUser$ = this.user$.asObservable();

  login(user: User) { this.user$.next(user); }
  logout()          { this.user$.next(null); }
  getUser()         { return this.user$.getValue(); }
}

// In component
user$ = this.authSvc.currentUser$;
// Template: {{ user$ | async }}
⚠️ Always expose as .asObservable() to prevent external code from calling .next() directly — encapsulation.
Q21 + Q22 — Lifecycle Hooks

Angular Lifecycle Hooks

Angular calls these methods in a fixed order during the life of a component/directive.

1
constructor()
DI injection. No DOM. No inputs set yet.
2
ngOnChanges()
Called when @Input values change. Gets SimpleChanges.
3
ngOnInit()
Component initialised. Inputs ready. Fetch data here.
4
ngDoCheck()
Custom change detection. Runs every CD cycle. Use carefully.
5
ngAfterContentInit()
After <ng-content> is projected.
6
ngAfterContentChecked()
After projected content checked.
7
ngAfterViewInit()
Child views initialised. DOM ready. Use @ViewChild here.
8
ngAfterViewChecked()
After view checked. Runs on every CD cycle.
9
ngOnDestroy()
Cleanup: unsubscribe, clear intervals, remove listeners.

ngAfterViewInit — Use Cases

ngAfterViewInit(): void {
  // 1. Access DOM element via ViewChild
  this.inputRef.nativeElement.focus();

  // 2. Initialise third-party libraries
  Chart(this.chartRef.nativeElement, this.chartConfig);

  // 3. Measure DOM element dimensions
  const width = this.containerRef.nativeElement.offsetWidth;
}

ngOnDestroy — Always Clean Up

private destroy$ = new Subject<void>();

ngOnInit() {
  this.data$.pipe(
    takeUntil(this.destroy$)  // auto-unsubscribe
  ).subscribe(...)
}

ngOnDestroy() {
  this.destroy$.next();
  this.destroy$.complete();
}
⚠️ Not unsubscribing = memory leak. Always use takeUntil, async pipe, or takeUntilDestroyed() (Angular 16+).
Q23 — Performance Optimization

Angular Performance Optimization

10 years = you should know all of these. Mention at least 5 with real project context.

1. OnPush Change Detection

Only re-renders when @Input reference changes or Observable emits.

changeDetection: ChangeDetectionStrategy.OnPush
2. Lazy Loading Modules

Split bundles by route. Reduces initial load time significantly.

3. TrackBy in *ngFor

Prevents re-rendering unchanged list items.

*ngFor="let u of users; trackBy: trackByFn"
trackByFn(i, u) { return u.id; }
4. Async Pipe

Auto-subscribes and auto-unsubscribes. No memory leaks.

{{ users$ | async }}
5. Pure Pipes

Only recalculates when input reference changes. Default pipe behaviour.

6. Virtual Scrolling

CDK ScrollingModule renders only visible rows in large lists.

7. Debounce User Inputs

Reduce API calls on search inputs.

debounceTime(300), distinctUntilChanged()
8. AOT Compilation

Compiles templates at build time. Default in prod. Faster startup.

9. Tree-shaking & Bundle Analysis

Removes unused code. Use webpack-bundle-analyzer.

🎯 Senior Answer: "In my last project I reduced the initial bundle from 2.1MB to 380KB using lazy loading + tree-shaking, and cut re-render time 60% by switching 40 components to OnPush strategy."
Q24 — Routing

Routing in Angular

Route Configuration

const routes: Routes = [
  { path: '',       redirectTo: 'home', pathMatch: 'full' },
  { path: 'home',    component: HomeComponent },
  { path: 'user/:id', component: UserComponent },
  {
    path: 'admin',
    canActivate: [AuthGuard],       // Route guard
    loadChildren: () => import('./admin/admin.module')
      .then(m => m.AdminModule)
  },
  { path: '**', component: NotFoundComponent }
];

Route Guards

@Injectable({ providedIn: 'root' })
export class AuthGuard implements CanActivate {
  canActivate(): boolean {
    if (this.auth.isLoggedIn()) return true;
    this.router.navigate(['/login']);
    return false;
  }
}

Route Guards Types

GuardPurpose
canActivateBlock route access (auth check)
canActivateChildBlock child routes
canDeactivateWarn before leaving (unsaved changes)
canLoadPrevent lazy module download
resolvePre-fetch data before activating

Read Route Params

// Snapshot (one-time)
const id = this.route.snapshot.paramMap.get('id');

// Observable (updates on param change)
this.route.paramMap.subscribe(params => {
  const id = params.get('id');
});
💡 Angular 15+: Functional guards replace class-based guards — simpler and tree-shakeable.
canActivate: [() => inject(AuthService).isLoggedIn()]
Q25 + Q26 + Q27 — Authentication

Authentication — Login API, Types & Verification

Login API Flow (JWT)

// 1. Send credentials
login(creds: Credentials): Observable<AuthResponse> {
  return this.http.post<AuthResponse>(
    '/api/auth/login', creds
  ).pipe(
    tap(res => {
      // 2. Store token securely
      localStorage.setItem('token', res.accessToken);
      this.user$.next(res.user);
    })
  );
}

// 3. Attach token via HTTP Interceptor
intercept(req, next) {
  const token = localStorage.getItem('token');
  const cloned = req.clone({
    headers: req.headers.set('Authorization', `Bearer ${token}`)
  });
  return next.handle(cloned);
}

Types of Authentication

TypeHowUse case
JWTToken in header/localStorageSPA + REST API
OAuth23rd-party auth (Google, MS)SSO, social login
SessionCookie + server sessionTraditional web
SAMLXML assertionsEnterprise SSO
API KeyKey in headerB2B, microservices

Verifying User Data

  1. Backend validates JWT signature using secret key
  2. Checks token expiry (exp claim)
  3. Extracts user ID from payload, loads from DB
  4. Frontend: Route guards check token before navigation
  5. Handle 401 responses globally via interceptor → redirect to login
⚠️ Security: Never store sensitive data in JWT payload — it's base64-decoded, not encrypted. Use httpOnly cookies for refresh tokens.
Q28 — HTTP Verbs

HTTP Verbs with Angular HttpClient

VerbPurposeBody?Idempotent?Angular Example
GETRetrieve data❌✅http.get('/api/users')
POSTCreate resource✅❌http.post('/api/users', body)
PUTReplace full resource✅✅http.put('/api/users/1', body)
PATCHPartial update✅✅http.patch('/api/users/1', partial)
DELETERemove resource❌✅http.delete('/api/users/1')
// Complete CRUD service pattern
@Injectable({ providedIn: 'root' })
export class UserService {
  private url = '/api/users';

  getAll()               { return this.http.get<User[]>(this.url); }
  getById(id: number)     { return this.http.get<User>(`${this.url}/${id}`); }
  create(u: User)         { return this.http.post<User>(this.url, u); }
  update(id, u: User)    { return this.http.put<User>(`${this.url}/${id}`, u); }
  patch(id, p: Partial<User>) { return this.http.patch<User>(`${this.url}/${id}`, p); }
  delete(id: number)     { return this.http.delete(`${this.url}/${id}`); }
}
💡 PUT vs PATCH:
PUT replaces the entire resource — missing fields become null.
PATCH updates only the provided fields — partial update.
⚠️ Idempotent = calling the same request multiple times gives the same result. GET, PUT, PATCH, DELETE are idempotent. POST is not (creates a new resource each time).
✅ Always handle errors with catchError in a shared interceptor — don't handle in every component.
