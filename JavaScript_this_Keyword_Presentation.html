01 — Global Context

Global Context

When this is used outside of any function or object, it refers to the global object.

Browser

The global object is window.

// Browser
console.log(this);       // Window {...}
console.log(this === window); // true

Node.js

The global object is global, but at module scope this is {} (the module's exports).

// Node.js (CommonJS module)
console.log(this);      // {} (module.exports)
console.log(global);   // [Object: global]
💡 globalThis (ES2020) is the universal cross-environment reference to the global object — works in browser, Node.js, and workers. 
// Browser – top-level script
console.log(this);   // Window

var x = 42;
console.log(this.x); // 42 (var leaks to window)

let y = 99;
console.log(this.y); // undefined (let doesn't)

// ES2020+
console.log(globalThis); // Window (browser)
                           // global  (Node.js)
                           // self    (Worker)
⚠️ Gotcha: var at global scope attaches to window, but let and const do not.
02 — Function Context

Function Context

Behavior of this inside a regular function depends on strict mode.

Non-Strict Mode

Called a default binding. this is the global object.

function show() {
  console.log(this);
}

show(); // Window (browser)
        // global (Node.js)
Default binding: When no other rule applies, non-strict this falls back to the global object.

Strict Mode

this is undefined — prevents accidental global pollution.

"use strict";

function show() {
  console.log(this);
}

show(); // undefined

// Also applies inside class bodies (always strict)
class Foo {
  bar() { console.log(this); }
}
const fn = new Foo().bar;
fn(); // undefined (strict by default in class)
💡 All class bodies in ES6 run in strict mode automatically, so method extraction always yields undefined for this.
03 — Method Context

Method Context (Inside an Object)

When called as an object method, this refers to the object to the left of the dot (implicit binding).

const person = {
  name: 'Alice',
  greet() {
    console.log(`Hi, I'm ${this.name}`);
  }
};

person.greet(); // "Hi, I'm Alice"

// ⚠️ Method extraction LOSES context
const g = person.greet;
g(); // "Hi, I'm undefined"  (strict: TypeError)
⚠️ Implicit binding lost: Storing a method in a variable and calling it removes the object context.

Nested Object

const outer = {
  name: 'outer',
  inner: {
    name: 'inner',
    show() {
      console.log(this.name);
    }
  }
};

outer.inner.show(); // "inner"
// this = the object IMMEDIATELY left of the dot
💡 Rule: this is the object that owns the call — the one directly before the final dot.
04 — Arrow Functions

Arrow Functions & Lexical this

Arrow functions do not have their own this. They capture this from the enclosing lexical scope at definition time.

Correct Use — Preserving Context

const timer = {
  count: 0,
  start() {
    setInterval(() => {     // arrow: this = timer
      this.count++;
      console.log(this.count);
    }, 1000);
  }
};
timer.start(); // 1, 2, 3 ... ✓

Wrong Use — Arrow in Object Literal

const person = {
  name: 'Bob',
  // ❌ arrow captures `this` from the
  //    module/global scope, NOT the object
  greet: () => {
    console.log(this.name);
  }
};
person.greet(); // undefined ❌
❌ Never use arrow functions as object literal methods when you need this to refer to the object.
💡 Arrow functions also cannot be used as constructors (new ArrowFn() throws a TypeError) and cannot be re-bound with .call(), .apply(), or .bind().
05 — Event Handlers

Event Handlers

In a DOM event handler, this refers to the element that fired the event.

Regular Function Handler

// HTML: <button id="btn">Click Me</button>

document
  .getElementById('btn')
  .addEventListener('click', function() {
    console.log(this);       // <button> element
    console.log(this.id);    // "btn"
  });

// Inline handler (same rule)
// <button onclick="console.log(this)">
// logs the button element

Arrow Function Handler

// Arrow: this comes from outer scope
document
  .getElementById('btn')
  .addEventListener('click', () => {
    console.log(this); // Window! (not button)
  });

// Use event.currentTarget instead
document
  .getElementById('btn')
  .addEventListener('click', (e) => {
    console.log(e.currentTarget); // button ✓
  });
💡 When using arrow functions in event handlers (common in React-style patterns), access the element via event.currentTarget or event.target rather than relying on this.
06 — Constructor Functions

Constructor Functions & new Binding

When a function is invoked with new, JavaScript creates a brand-new object and sets this to that object — called new binding.

function Person(name, age) {
  this.name = name;  // bound to new object
  this.age  = age;
  this.greet = function() {
    return `Hi, I'm ${this.name}`;
  };
}

const alice = new Person('Alice', 30);
console.log(alice.greet());  // "Hi, I'm Alice"
console.log(alice.age);      // 30

What new does internally

  1. Creates a new empty object: {}
  2. Sets [[Prototype]] to Person.prototype
  3. Binds this to that new object
  4. Executes the constructor body
  5. Returns the new object (unless the constructor explicitly returns an object)
⚠️ If a constructor returns a plain object explicitly, this is discarded and that object is returned instead. Primitives are ignored.
07 — Class Context

Class Context

In ES6 classes, this refers to the instance. Class bodies run in strict mode automatically.

class Animal {
  constructor(name) {
    this.name = name;
  }

  speak() {
    return `${this.name} makes a noise.`;
  }
}

class Dog extends Animal {
  speak() {
    return `${this.name} barks.`;
  }
}

const d = new Dog('Rex');
console.log(d.speak()); // "Rex barks."

Class Fields — Arrow vs Regular

class Counter {
  count = 0;

  // Arrow: this is FIXED to instance ✓
  increment = () => { this.count++; };

  // Regular: this can be lost ⚠️
  decrement() { this.count--; }
}

const c = new Counter();
const { increment, decrement } = c;

increment(); // ✓ works — c.count = 1
decrement(); // ❌ TypeError: this is undefined
💡 Class field arrow functions are a common React pattern to avoid binding in constructors. Each instance gets its own copy of the function.
08 — Explicit Binding

call / apply / bind

These methods let you set this explicitly — called explicit binding.

call

Invokes immediately. Arguments passed individually.

function greet(city) {
  return `${this.name}, ${city}`;
}
const u = { name: 'Alice' };

greet.call(u, 'NYC');
// "Alice, NYC"
apply

Invokes immediately. Arguments passed as an array.

greet.apply(u, ['LA']);
// "Alice, LA"

// Useful for variadic spread:
Math.max.apply(null, [1,5,3]);
// 5
bind

Returns a new function with this permanently set.

const bound = greet.bind(u);
bound('SF'); // "Alice, SF"

// Partial application:
const nyc = greet.bind(u, 'NYC');
nyc(); // "Alice, NYC"
call(thisArg, arg1, arg2, ...)
Immediately invoked. Spreads args.
apply(thisArg, [args])
Immediately invoked. Array args.
bind(thisArg, ...partials)
Returns new function. Supports partial application.
09 — Binding Priority Rules ✦ Added

The 4 Rules & Their Priority

When multiple rules could apply, JavaScript follows a strict precedence order.

1. new Binding
›
2. Explicit Binding
call / apply / bind
›
3. Implicit Binding
obj.method()
›
4. Default Binding
global / undefined

bind beats call

function who() { return this.name; }
const bound = who.bind({ name: 'Alice' });

// call can't override a hard bind:
console.log(bound.call({ name: 'Bob' }));
// "Alice"  ← bind wins!

new beats bind

function Foo(x) { this.x = x; }
const BoundFoo = Foo.bind({ x: 999 });

const obj = new BoundFoo(42);
console.log(obj.x); // 42  ← new wins!

10 — ES Modules & globalThis ✦ Added

ES Modules & globalThis

Two modern behaviors that trip up experienced developers.

this at Top Level in ES Modules

<!-- index.html -->
<script type="module" src="app.js"></script>

// app.js — ES module
console.log(this);
// undefined  ← NOT window!

// Compare: classic script
<script src="app.js"></script>
console.log(this);
// Window  ← global as expected
⚠️ ES modules are implicitly in strict mode. Top-level this is undefined, not window.

globalThis — ES2020

// Works everywhere: browser, Node.js, worker
console.log(globalThis);

// Before globalThis, developers wrote:
const getGlobal = () => {
  if (typeof window !== 'undefined') return window;
  if (typeof global !== 'undefined') return global;
  if (typeof self   !== 'undefined') return self;
};

// Now just:
console.log(globalThis); // Window | global | self

this in Getters & Setters

const circle = {
  radius: 5,
  get area() {
    return Math.PI * this.radius ** 2;
  }
};
console.log(circle.area); // 78.53...
11 — Common Pitfalls

Common Pitfalls

Pitfall 1

Losing this in Callbacks

const obj = {
  name: 'Alice',
  greet() {
    setTimeout(function() {
      console.log(this.name); // ❌ undefined
    }, 100);
  }
};
obj.greet();

// Fix: use arrow function
setTimeout(() => console.log(this.name), 100); // ✓
❌ Why it breaks: setTimeout invokes the callback as a plain function call — not as obj.something(). So Default Binding kicks in: this becomes window (or undefined in strict mode).

✅ Why the fix works: Arrow functions have no own this. They inherit this lexically from where they are written — inside greet(), where this is obj. That value is captured at definition time and never changes.
Pitfall 2

Arrow Function as Object Method

const person = {
  name: 'Bob',
  greet: () => {
    // ❌ this is NOT person, it's outer scope
    console.log(this.name); // undefined
  }
};
person.greet();

// Fix: use shorthand method
greet() { console.log(this.name); } // ✓
❌ Why it breaks: Arrow functions capture this from their enclosing lexical scope. Here the arrow is defined at the top level (outside any function), so this = window / undefined — never person, even when called as person.greet().

✅ Why the fix works: A shorthand method (greet() {}) is a regular function. It follows Implicit Binding — this is set to the object to the left of the dot (person) at call time.
Pitfall 3

Method Extraction

class Timer {
  constructor() { this.ticks = 0; }
  tick() { this.ticks++; }
}
const t = new Timer();
// ❌ setInterval gets a bare function ref
setInterval(t.tick, 1000);

// Fix options:
setInterval(() => t.tick(), 1000);        // ✓
setInterval(t.tick.bind(t), 1000);         // ✓
❌ Why it breaks: t.tick passes only the function reference — the link to t is lost. When setInterval calls it later, there's no receiver, so Default Binding applies: this becomes window / undefined. this.ticks++ modifies nothing on the instance.

✅ Fix 1 — Arrow wrapper: () => t.tick() calls the method on t each time, so Implicit Binding applies.
✅ Fix 2 — bind: t.tick.bind(t) hard-binds this to t permanently (Explicit Binding).
Pitfall 4

Nested Regular Function

const obj = {
  val: 42,
  outer() {
    function inner() {
      return this.val; // ❌ this is global
    }
    return inner();
  }
};
console.log(obj.outer()); // undefined

// Fix: arrow function
const inner = () => this.val; // ✓
❌ Why it breaks: Even though inner is defined inside outer(), it is called as a plain function — not as obj.inner(). Every regular function gets its own this based on how it is called. Plain call → Default Binding → this = window / undefined.

✅ Why the fix works: Replacing function inner() with an arrow const inner = () => means inner has no own this. It looks up to the enclosing outer() scope, where this = obj (set by Implicit Binding when obj.outer() was called).

Output-Based Questions Q1 – Q5

Q1 — Method Extraction
const person = { name: 'Alice', getName() { return this.name; } };
const fn = person.getName;
console.log(fn());
Output
undefined (strict mode: TypeError: Cannot read properties of undefined)
Reason: fn is called without a receiver — default binding applies.
Q2 — bind vs call
function greet() { return this.name; }
const bound = greet.bind({ name: 'Alice' });
console.log(bound.call({ name: 'Bob' }));
Output
"Alice"
Reason: bind creates a hard binding. Calling .call() on a bound function cannot override the bound this.
Q3 — Arrow in setTimeout
const obj = {
  name: 'Alice',
  greet() {
    setTimeout(function() { console.log(this.name); }, 0);
  }
};
obj.greet();
Output
undefined (browser non-strict) / TypeError (strict)
Reason: The setTimeout callback is a regular function. Its this defaults to window (or is undefined in strict mode).
Q4 — new beats bind
function Foo(val) { this.val = val; }
const Bound = Foo.bind({ val: 999 });
const obj = new Bound(42);
console.log(obj.val);
Output
42
Reason: new has higher priority than bind. The bound object { val: 999 } is ignored; a fresh instance is created.
Q5 — Nested Object Access
const a = { x: 1, b: { x: 2, get() { return this.x; } } };
console.log(a.b.get());
Output
2
Reason: this = the object immediately to the left of the last dot = b. So this.x = 2.

Summary

Key Takeaways

Rule 1

It's all about the call site. this is determined when a function is called, not when it's defined — except for arrow functions.

Rule 2

Arrow functions inherit this lexically. They capture it from the enclosing scope at definition time and it cannot be changed.

Rule 3

Binding priority: new > explicit (bind/call/apply) > implicit (obj.fn()) > default (fn()).

Rule 4

Strict mode and ES modules set this to undefined in default binding scenarios — always prefer explicit binding.

Rule 5

Use class field arrows to permanently bind this to an instance — ideal for event callbacks and React component handlers.

Rule 6

globalThis is the safe cross-environment way to access the global object. Top-level this in an ES module is undefined.

🚀 Mastering this = mastering JavaScript's execution context model
13 — Object Literal & Scope ✦ Deep Dive

What is an Object Literal? Why it matters for this

Object Literal = Directly Written Object

You literally write the object in code using { }. Just like "hello" is a string literal, { } is an object literal.

// String literal
const str = "hello";

// Number literal
const n = 42;

// Object literal — same idea
const person = {
  name: 'Alice',
  age: 30,
  greet() { console.log('Hi!'); }
};
Other ways to create objects: new Fn(), Object.create(), new MyClass() — but none are as concise as the literal.

The Critical Rule: Object Literals Don't Create this Scope

Only functions and classes create a new this scope. A { } object literal does NOT.

// Object literal — NO new this scope
const person = {
  name: 'Bob',

  // ❌ Arrow captures outer this (window/undefined)
  //    because object literal is NOT a scope
  greet: () => {
    console.log(this.name); // undefined
  },

  // ✅ Regular method: this = person at call time
  greetOk() {
    console.log(this.name); // "Bob"
  }
};

14 — Explicit vs Implicit Binding ✦ Deep Dive

Explicit vs Implicit Binding

Two of the four binding rules. Understanding the difference is essential for senior-level JS.

Implicit Binding — JS decides automatically

this = object to the left of the dot at call time. You never said what this is — JS figured it out.

const car = {
  brand: 'Toyota',
  drive() {
    return this.brand; // implicitly = car
  }
};
console.log(car.drive()); // "Toyota"

// ⚠️ Implicit binding can be LOST
const fn = car.drive;
fn(); // undefined — no object before dot!
⚠️ Implicit binding is fragile — the moment you extract the method from the object, the binding is lost.

Explicit Binding — You decide

You directly tell JS: "use THIS as this" via call, apply, or bind.

function drive() {
  return this.brand;
}

const car   = { brand: 'Toyota' };
const bike  = { brand: 'Honda'  };

// Reuse same function with different contexts
drive.call(car);   // "Toyota"
drive.call(bike);  // "Honda"

// Permanent bind
const driveCar = drive.bind(car);
driveCar(); // always "Toyota"
✅ Explicit binding is reliable — you always know exactly what this will be.
Implicit — Who decides: JavaScript | How: Object before dot | Risk: Can be lost
Explicit — Who decides: You | How: call/apply/bind | Risk: None (hard-bound)
15 — this in Array Methods ✦ Deep Dive

this in Array Methods

forEach, map, filter, find, some, every — all accept an optional thisArg second parameter.

Without thisArg — this is lost

const processor = {
  multiplier: 3,
  process(arr) {
    // ❌ regular function: this = undefined (strict)
    return arr.map(function(n) {
      return n * this.multiplier; // TypeError!
    });
  }
};

Fix Option 1 — Arrow Function

process(arr) {
  // ✅ arrow inherits this from process()
  return arr.map(n => n * this.multiplier);
}
// Result: [3, 6, 9] for [1, 2, 3]

Fix Option 2 — thisArg parameter

process(arr) {
  // ✅ pass `this` as second arg to map
  return arr.map(function(n) {
    return n * this.multiplier;
  }, this); // ← thisArg
}

Array Methods with thisArg

MethodthisArg Position
arr.forEach(fn, thisArg)2nd arg
arr.map(fn, thisArg)2nd arg
arr.filter(fn, thisArg)2nd arg
arr.find(fn, thisArg)2nd arg
arr.some(fn, thisArg)2nd arg
arr.every(fn, thisArg)2nd arg
arr.reduce(fn)❌ No thisArg
⚠️ thisArg is ignored if you pass an arrow function — arrows don't bind this.
16 — this in Promises & async/await ✦ Deep Dive

this in Promises & async/await

Async code is one of the most common places where this gets lost. Knowing the patterns is essential.

Problem — this lost in .then()

class DataService {
  constructor() { this.data = []; }

  fetchData() {
    fetch('/api/data')
      .then(function(res) { return res.json(); })
      .then(function(json) {
        // ❌ this is undefined (strict class context)
        this.data = json;
      });
  }
}

Fix — Arrow in .then()

fetchData() {
  fetch('/api/data')
    .then(res => res.json())
    .then(json => {
      // ✅ arrow: this = DataService instance
      this.data = json;
    });
}

async/await — this is preserved ✅

class DataService {
  constructor() { this.data = []; }

  // ✅ async method: this works perfectly
  async fetchData() {
    const res  = await fetch('/api/data');
    const json = await res.json();
    this.data = json; // ✅ this = instance
  }
}
💡 async/await preserves this correctly because the async keyword makes the method act like a regular method — this is set by the call site.

⚠️ Detach async method = same problem

const svc = new DataService();
const fn = svc.fetchData; // extracted
await fn(); // ❌ this is undefined
await svc.fetchData(); // ✅ this = svc
17 — this in IIFE ✦ Deep Dive

this in IIFE (Immediately Invoked Function Expressions)

An IIFE is called immediately at definition — with no object prefix, so default binding applies.

What is an IIFE?

// Regular function — called later
function greet() { console.log('hi'); }
greet();

// IIFE — called immediately, no name
(function() {
  console.log('hi'); // runs instantly
})();

// Arrow IIFE
(() => {
  console.log('hi');
})();
IIFEs were heavily used before ES6 modules to avoid polluting the global scope. Still seen in legacy code.

this inside IIFE

// Non-strict: this = window
(function() {
  console.log(this); // Window
})();

// Strict: this = undefined
(function() {
  'use strict';
  console.log(this); // undefined
})();

// Inside a method — regular IIFE loses this
const obj = {
  val: 42,
  run() {
    (function() {
      console.log(this.val); // ❌ undefined
    })();

    // Fix: arrow IIFE inherits this
    (() => {
      console.log(this.val); // ✅ 42
    })();
  }
};
obj.run();
18 — this & Prototype Chain ✦ Deep Dive

this & the Prototype Chain

When a method is found on the prototype, this still refers to the original instance — not the prototype object.

Prototype Method — this = Instance

function Animal(name) {
  this.name = name;
}

Animal.prototype.speak = function() {
  // this = the instance, NOT Animal.prototype
  return this.name + ' speaks';
};

const dog = new Animal('Rex');
console.log(dog.speak()); // "Rex speaks"
// JS looks up speak on prototype,
// but calls it with dog as `this`
💡 The prototype lookup chain finds the method, but the call site decides this. The object used to call is always the receiver.

Inheritance Chain — this is always the leaf instance

class Vehicle {
  type() { return this.brand; }
}

class Car extends Vehicle {
  constructor() {
    super();
    this.brand = 'Toyota';
  }
}

const c = new Car();
console.log(c.type()); // "Toyota"
// type() is on Vehicle.prototype
// but this = c (Car instance)

Object.create() — same rule

const proto = {
  greet() { return this.name; }
};
const obj = Object.create(proto);
obj.name = 'Alice';
console.log(obj.greet()); // "Alice"
// greet is on proto, this = obj
19 — All Ways to Lose this ✦ Deep Dive

All Ways to Lose this — and How to Fix Each

Way 1

Method Extraction / Destructuring

const { greet } = person; greet(); // ❌
// Fix:
const greet = person.greet.bind(person); // ✅
Way 2

Regular Function in setTimeout

setTimeout(function() { this.run(); }, 0); // ❌
// Fix:
setTimeout(() => { this.run(); }, 0); // ✅
Way 3

Callback passed to another function

emitter.on('event', obj.handler); // ❌
// Fix:
emitter.on('event', obj.handler.bind(obj)); // ✅
emitter.on('event', (...a) => obj.handler(...a)); // ✅
Way 4

Arrow as Object Method

const o = { fn: () => this }; // ❌ outer this
// Fix:
const o = { fn() { return this; } }; // ✅
Way 5

Nested Regular Function Inside Method

outer() {
  function inner() { return this.x; } // ❌
  // Fix 1 — arrow:
  const inner = () => this.x; // ✅
  // Fix 2 — self/that pattern:
  const self = this; function inner() { return self.x; } // ✅
}
Way 6

Regular fn in .then() / array methods

arr.map(function(x) { return this.mul * x; }); // ❌
// Fix 1 — arrow:
arr.map(x => this.mul * x); // ✅
// Fix 2 — thisArg:
arr.map(function(x) { return this.mul * x; }, this); // ✅
🎯 Universal Fix: Whenever you need to pass a method as a callback and it uses this — use either an arrow wrapper, .bind(), or a class field arrow.
20 — Decision Flowchart ✦ Deep Dive

How to Determine this — Decision Tree

When you see this anywhere in code, ask these questions in order:

1
Is it an arrow function? → Yes: this = enclosing scope's this. Stop here. Ignore all other rules.
2
Is it called with new? → Yes: this = the newly created object.
3
Is it called with .call(), .apply(), or .bind()? → Yes: this = the explicitly passed context.
4
Is there an object before the dot? e.g. obj.fn() → Yes: this = that object (implicit binding).
5
None of the above — plain call fn()? → Default binding: window (non-strict) or undefined (strict / ES module / class).
Special cases that override everything:
• ES Module top-level → always undefined
• Class body → always strict mode
• Bound function (.bind) → cannot be re-bound by call/apply

// 1. Arrow?        → lexical this
// 2. new?          → new object
// 3. call/apply/bind? → passed ctx
// 4. obj.fn()?     → obj
// 5. fn()?         → window / undefined

// Look at the CALL SITE, not the definition.
